# L2 — Detailed Requirements

## Raffle Web Application

Each requirement traces to an L1 parent and includes specific acceptance criteria.

---

## Public Raffle Drawing

### L2-001: Draw Name Action

**Traces to:** L1-001, L1-011

The public raffle page shall consist of four elements only: a heading, an optional subheading, a name display area, and a "Draw" button. When the button is activated, it randomly selects one undrawn name from the active raffle and reveals it through the configured animation sequence (see L2-031, L2-032) with an accompanying sound effect (see L2-037).

**Acceptance Criteria:**

- The page displays the raffle heading prominently at the top.
- The optional subheading is displayed beneath the heading when provided.
- A name display area is centered on the page, showing the last drawn name or a placeholder before the first draw.
- A clearly labeled "Draw" button is visible below the name display.
- Clicking the button selects exactly one name at random from the pool of undrawn names.
- The drawn name is revealed through the name cycling animation (L2-031) with sound (L2-037), followed by the celebratory reveal (L2-032).
- The button is disabled during the animation sequence and when no names remain to be drawn.
- A clear message is displayed in the name area when all names have been drawn.
- No draw history, remaining count, or other ancillary information is displayed on the public page.

---

### L2-002: No Active Raffle State

**Traces to:** L1-001, L1-004

When no raffle is currently set to active, the public page shall display an informational message indicating that no raffle is available.

**Acceptance Criteria:**

- The page displays a user-friendly message such as "No raffle is currently active."
- The "Draw" button is not rendered when no raffle is active.
- No error is thrown; the page loads cleanly.

---

## Draw Animation and Visual Experience

### L2-031: Name Cycling Animation

**Traces to:** L1-011, L1-001

Upon triggering a draw, the public raffle page shall display a smooth animation that rapidly cycles through candidate names from the raffle's participant pool, then gradually decelerates until coming to a complete stop on the selected winner's name.

**Acceptance Criteria:**

- After the user clicks "Draw," the display area immediately begins cycling through names at a rapid pace.
- The cycling speed decreases progressively over the duration of the animation, creating a natural deceleration effect.
- The animation concludes by landing precisely on the server-determined winner's name.
- The total animation duration is between 3 and 6 seconds (tuned per animation style).
- The animation is smooth and fluid, maintaining a consistent frame rate without visible jank or stutter.
- The draw button is disabled for the duration of the animation to prevent concurrent triggers.
- The names displayed during cycling are drawn from the current raffle's participant list (not placeholder text).

---

### L2-032: Celebratory Winner Reveal Animation

**Traces to:** L1-011

When the name cycling animation concludes and the winner is revealed, the page shall play a celebratory visual animation to highlight the result.

**Acceptance Criteria:**

- A celebratory effect (e.g., confetti burst, fireworks, sparkles, or equivalent) plays immediately when the winner's name is revealed.
- The winner's name is prominently displayed and visually emphasized (e.g., enlarged, highlighted, or animated entrance).
- The celebration animation lasts between 2 and 5 seconds and does not obstruct the winner's name.
- The celebration does not block user interaction indefinitely; the user can trigger another draw (if names remain) after the animation completes.
- The animation respects the user's operating system reduced-motion preference (`prefers-reduced-motion`); when enabled, the celebration is replaced with a static visual emphasis on the winner's name.

---

### L2-033: Animation Accessibility

**Traces to:** L1-011, L1-008

All draw and celebratory animations shall be accessible and shall not cause harm to users with motion sensitivities or who rely on assistive technologies.

**Acceptance Criteria:**

- When `prefers-reduced-motion: reduce` is active, the name cycling animation is replaced with a brief fade-in or instant reveal of the winner.
- Screen readers announce the winner's name via an ARIA live region after the animation completes, regardless of animation style.
- Animations do not flash more than three times per second (WCAG 2.3.1 compliance).
- The winner result is available in the DOM for assistive technologies even during animation playback.

---

### L2-037: Draw Cycling Sound Effect

**Traces to:** L1-011

During the name cycling animation, the system shall play a fun, rhythmic sound effect that matches the pace of the cycling names and decelerates in sync with the animation.

**Acceptance Criteria:**

- A sound effect begins playing when the name cycling animation starts.
- The sound rhythm matches the speed of name changes (e.g., a tick, click, or drum roll effect).
- The sound decelerates naturally as the cycling slows down, staying in sync with the visual animation.
- A distinct celebratory sound plays when the winner is revealed (e.g., a chime, fanfare, or applause).
- Sound respects the device volume settings and does not auto-play at an unexpectedly loud level.
- A mute/unmute toggle is available on the public page for users who prefer silence.
- Sound is muted by default to comply with browser autoplay policies; the user must interact with the page (e.g., click "Draw") before sound plays.
- The mute preference persists across draws within the same session.

---

## Unique Draw Enforcement

### L2-004: Prevent Duplicate Draws

**Traces to:** L1-002

The system shall remove each drawn name from the available pool immediately upon draw, ensuring no name can be selected more than once per raffle.

**Acceptance Criteria:**

- A drawn name never appears in subsequent draw results.
- The available-name count decreases by one after each draw.
- Concurrent draw requests do not result in the same name being drawn twice.

## Back Office — Raffle Management

### L2-006: Create Raffle

**Traces to:** L1-003, L1-011

An administrator shall be able to create a new raffle by providing a raffle name, a list of participant names, a page heading and optional subheading, a visual theme, and an animation style.

**Acceptance Criteria:**

- A form is provided with fields for raffle name, participant names, heading, subheading, theme selection, and animation style selection.
- The raffle name is required and must be unique across all raffles.
- Participant names can be entered one per line or as a comma-separated list.
- Duplicate participant names within a single raffle are rejected with a validation message.
- Heading, subheading, theme, and animation style fields conform to L2-034, L2-035, and L2-036.
- The raffle is saved and appears in the raffle list upon successful creation.

---

### L2-034: Raffle Page Heading and Subheading

**Traces to:** L1-003, L1-011

When creating or editing a raffle, an administrator shall be able to provide a custom heading and subheading that are displayed on the public-facing raffle page for that raffle.

**Acceptance Criteria:**

- The raffle creation and edit forms include fields for heading and subheading.
- The heading is required; the subheading is optional.
- The heading is displayed prominently at the top of the public raffle page when that raffle is active.
- The subheading, if provided, is displayed beneath the heading.
- Heading and subheading support plain text only (no HTML or markdown).
- Heading is limited to 100 characters; subheading is limited to 250 characters.
- Input is validated and sanitized on both client and server.

---

### L2-035: Raffle Theme Selection

**Traces to:** L1-003, L1-011

When creating or editing a raffle, an administrator shall be able to select a visual theme that controls the look and feel of the public-facing raffle page for that raffle.

**Acceptance Criteria:**

- The raffle creation and edit forms include a theme selector.
- At least three distinct themes are available (e.g., Classic, Festive, Corporate).
- Each theme defines a cohesive set of colors, typography, and background styling for the public raffle page.
- A visual preview or thumbnail of each theme is shown in the selector to aid the administrator's choice.
- A default theme is applied if no explicit selection is made.
- All themes meet WCAG 2.2 AA contrast requirements (per L2-024).
- Changing the theme of an active raffle is immediately reflected on the public page.

---

### L2-036: Raffle Animation Style Selection

**Traces to:** L1-003, L1-011

When creating or editing a raffle, an administrator shall be able to select the animation style used for the name cycling and celebratory reveal on the public-facing raffle page.

**Acceptance Criteria:**

- The raffle creation and edit forms include an animation style selector.
- At least three distinct animation styles are available (e.g., Slot Machine, Wheel Spin, Card Flip).
- Each animation style defines both the name cycling behavior and the corresponding celebratory reveal effect.
- A brief preview or description of each animation style is provided in the selector.
- A default animation style is applied if no explicit selection is made.
- All animation styles comply with the accessibility requirements defined in L2-033.
- Changing the animation style of an active raffle is immediately reflected on the public page.

---

### L2-007: Edit Raffle

**Traces to:** L1-003, L1-011

An administrator shall be able to edit the name, participant list, heading, subheading, theme, and animation style of an existing raffle. The participant list may only be edited if no names have been drawn; heading, subheading, theme, and animation style may be edited at any time.

**Acceptance Criteria:**

- The edit form is pre-populated with all current raffle settings.
- Changes are saved upon form submission.
- Participant list editing is disabled for raffles that have already had one or more names drawn; a message explains why.
- Heading, subheading, theme, and animation style can be changed regardless of draw status.
- Changes to theme, animation style, heading, or subheading on the active raffle are immediately reflected on the public page.
- Validation rules from L2-006, L2-034, L2-035, and L2-036 apply equally to edits.

---

### L2-008: Delete Raffle

**Traces to:** L1-003

An administrator shall be able to delete a raffle. Deleting the currently active raffle shall deactivate it.

**Acceptance Criteria:**

- A delete action is available for each raffle in the list.
- A confirmation dialog is presented before deletion.
- Upon deletion, the raffle and all associated draw history are permanently removed.
- If the deleted raffle was active, no raffle is active afterward.

---

### L2-009: Reset Raffle Draws

**Traces to:** L1-003, L1-002

An administrator shall be able to reset a raffle, clearing all draw history and returning all names to the available pool.

**Acceptance Criteria:**

- A reset action is available for raffles that have at least one drawn name.
- A confirmation dialog is presented before reset.
- After reset, all names are undrawn and the available count equals the total participant count.
- The draw history is cleared.

---

### L2-010: View Raffle Details

**Traces to:** L1-003

An administrator shall be able to view the full details of any raffle, including its name, participant list, draw history, and status (active/inactive, drawn count).

**Acceptance Criteria:**

- A detail view shows raffle name, total participant count, drawn count, and remaining count.
- The participant list is displayed with each name's drawn/undrawn status.
- Draw history is displayed in chronological order.

## Raffle Activation

### L2-011: Activate a Raffle

**Traces to:** L1-004

An administrator shall be able to set one raffle as the active raffle. Activating a raffle shall automatically deactivate any previously active raffle.

**Acceptance Criteria:**

- An "Activate" action is available for each inactive raffle.
- Activating a raffle immediately deactivates any other active raffle.
- Only one raffle can be active at any time; the system enforces this invariant.
- The active raffle is visually indicated in the raffle list.
- The public-facing page immediately reflects the newly active raffle.

---

### L2-012: Deactivate a Raffle

**Traces to:** L1-004

An administrator shall be able to deactivate the currently active raffle without activating another, resulting in no raffle being active.

**Acceptance Criteria:**

- A "Deactivate" action is available for the currently active raffle.
- After deactivation, no raffle is marked as active.
- The public-facing page displays the "no active raffle" state (per L2-002).

---

## Authentication and Authorization

### L2-013: Administrator Login

**Traces to:** L1-005

The back office application shall present a login page requiring a username/email and password.

**Acceptance Criteria:**

- A login form with email/username and password fields is presented.
- Successful authentication redirects the user to the back office dashboard.
- Failed authentication displays a generic error message that does not reveal whether the username or password was incorrect.
- The login form is protected against brute-force attacks via rate limiting.

---

### L2-014: Session Management

**Traces to:** L1-005, L1-007

Authenticated sessions shall be managed securely with automatic expiration.

**Acceptance Criteria:**

- Sessions expire after a configurable period of inactivity (default 30 minutes).
- Session tokens are stored securely (HttpOnly, Secure, SameSite cookies).
- A logout action is available and immediately invalidates the session.
- Expired sessions redirect the user to the login page with an informational message.

---

### L2-015: Protected Routes

**Traces to:** L1-005

All back office routes shall be inaccessible to unauthenticated users.

**Acceptance Criteria:**

- Unauthenticated requests to any back office route are redirected to the login page.
- Direct URL access to back office pages without authentication is blocked.
- API endpoints serving back office data return 401 Unauthorized for unauthenticated requests.

## Responsive Design

### L2-016: Public Page Responsiveness

**Traces to:** L1-006

The public raffle page shall render correctly and be fully functional across viewport breakpoints: xs (<576px), sm (576–767px), md (768–991px), lg (992–1199px), and xl (1200px+).

**Acceptance Criteria:**

- The heading, name display, and draw button are usable at all breakpoints.
- No horizontal scrolling occurs at any breakpoint.
- Touch targets meet a minimum size of 44x44 CSS pixels on small screens.
- Typography scales appropriately across breakpoints.
- The layout adjusts (e.g., single-column on xs/sm, multi-column where appropriate on lg/xl).

---

### L2-017: Back Office Responsiveness

**Traces to:** L1-006

The back office application shall render correctly and be fully functional across all viewport breakpoints defined in L2-016.

**Acceptance Criteria:**

- All administrative forms, tables, and actions are accessible at every breakpoint.
- Data tables adapt to small screens (e.g., horizontal scroll within container, card layout, or collapsible rows).
- Navigation is accessible via a hamburger menu or equivalent pattern on xs/sm screens.
- Modal dialogs and confirmation prompts are usable on all screen sizes.

## Security

### L2-018: Input Validation and Sanitization

**Traces to:** L1-007

All user input shall be validated on both client and server. Server-side validation is authoritative.

**Acceptance Criteria:**

- All form inputs are validated for type, length, and format on the server.
- Client-side validation provides immediate feedback but does not replace server-side checks.
- HTML and script injection attempts are neutralized (output encoding / escaping).
- SQL injection is prevented through parameterized queries or ORM usage.

---

### L2-019: HTTPS Enforcement

**Traces to:** L1-007

All communication between client and server shall occur over HTTPS.

**Acceptance Criteria:**

- HTTP requests are redirected to HTTPS.
- HSTS headers are set with a minimum max-age of one year.
- No mixed content (HTTP resources on HTTPS pages) is present.

---

### L2-020: CSRF Protection

**Traces to:** L1-007

All state-changing operations shall be protected against Cross-Site Request Forgery attacks.

**Acceptance Criteria:**

- Anti-CSRF tokens are included in all state-changing forms and API calls.
- Requests without valid CSRF tokens are rejected with a 403 Forbidden response.

---

### L2-021: Security Headers

**Traces to:** L1-007

The application shall set appropriate HTTP security headers on all responses.

**Acceptance Criteria:**

- Content-Security-Policy header restricts resource loading to trusted origins.
- X-Content-Type-Options is set to nosniff.
- X-Frame-Options is set to DENY (or SAMEORIGIN where framing is required).
- Referrer-Policy is set to strict-origin-when-cross-origin or more restrictive.
- Permissions-Policy restricts access to browser features not used by the application.

---

## Accessibility

### L2-022: Keyboard Navigation

**Traces to:** L1-008

All interactive elements shall be operable via keyboard alone.

**Acceptance Criteria:**

- All buttons, links, form controls, and dialogs are reachable via Tab/Shift+Tab.
- A visible focus indicator is present on the currently focused element.
- Focus order follows a logical reading order.
- Modal dialogs trap focus and return focus to the triggering element on close.
- No keyboard traps exist anywhere in the application.

---

### L2-023: Screen Reader Support

**Traces to:** L1-008

All content and interactive elements shall be accessible to screen readers.

**Acceptance Criteria:**

- All images and icons have appropriate alt text or aria-label attributes.
- Form fields have associated labels (via `<label>` or `aria-labelledby`).
- Dynamic content updates (e.g., draw results) are announced via ARIA live regions.
- Page structure uses semantic HTML (headings, landmarks, lists) appropriately.
- ARIA roles, states, and properties are used correctly where native semantics are insufficient.

---

### L2-024: Color and Contrast

**Traces to:** L1-008

The application shall meet WCAG 2.2 AA color contrast requirements.

**Acceptance Criteria:**

- Normal text has a minimum contrast ratio of 4.5:1 against its background.
- Large text (18pt or 14pt bold) has a minimum contrast ratio of 3:1.
- Interactive element states (focus, hover, active, disabled) are distinguishable without relying solely on color.
- Information is not conveyed by color alone; text labels or patterns supplement color cues.

The application shall follow a modular architecture with clear separation of concerns.

**Acceptance Criteria:**

## Reliability

### L2-028: Concurrent Draw Safety

**Traces to:** L1-010

The system shall handle concurrent draw requests without producing duplicate results or data corruption.

**Acceptance Criteria:**

- Simultaneous draw requests from multiple clients each receive a unique name.
- Database-level locking or atomic operations prevent race conditions.
- Under concurrent load, the total number of drawn names never exceeds the total participant count.

---

### L2-029: Data Persistence

**Traces to:** L1-010

All raffle data, including configuration and draw history, shall be persisted in a durable data store.

**Acceptance Criteria:**

- Raffle data survives application restarts.
- Draw results are persisted immediately upon draw (not deferred or batched).
- Data is backed up or stored in a system that supports recovery.

---

### L2-030: Error Handling

**Traces to:** L1-010

The application shall handle errors gracefully, providing meaningful feedback without exposing system internals.

**Acceptance Criteria:**

- Server errors return a generic user-friendly message; stack traces and internal details are never exposed to the client.
- Client-side errors (e.g., network failures) display actionable feedback (e.g., "Unable to connect. Please try again.").
- All errors are logged server-side with sufficient detail for diagnosis.
- The application recovers gracefully from transient failures (e.g., temporary database unavailability) without requiring a manual restart.

## QR Code Participant Entry

### L2-038: QR Code Display on Public Site

**Traces to:** L1-012

The public site shall include a dedicated page that displays a QR code linking to the participant self-registration page for the currently active raffle. The QR code shall be prominently displayed and large enough to scan reliably from a reasonable distance (e.g., projected on a screen).

**Acceptance Criteria:**

- Given a raffle is active and no names have been drawn, when a user navigates to the QR code page, then a QR code is displayed that encodes the URL of the participant entry page.
- Given no raffle is active, when a user navigates to the QR code page, then a user-friendly message is displayed (e.g., "No raffle is currently active") and no QR code is rendered.
- Given the active raffle has already had at least one name drawn, when a user navigates to the QR code page, then a message is displayed indicating that entries are closed and no QR code is rendered.
- The QR code shall be rendered at a minimum size of 256×256 CSS pixels on viewports ≥768px.
- The page shall include brief instructional text (e.g., "Scan to enter the raffle") adjacent to the QR code.
- The QR code shall be generated client-side and shall not depend on a third-party image service.

---

### L2-039: Participant Self-Registration Page

**Traces to:** L1-012

The system shall provide a public-facing participant entry page (the target of the QR code) where users can submit their name to be added to the active raffle. The page shall be optimized for mobile devices since the primary access path is scanning a QR code with a phone.

**Acceptance Criteria:**

- Given the entry page URL is opened on a mobile device and a raffle is active and accepting entries, when the page loads, then a form is displayed with a single text input for the participant's name and a "Join Raffle" submit button.
- Given the user enters a valid name and submits, when the server accepts the entry, then a confirmation message is displayed (e.g., "You're in! Good luck.") and the form is replaced by the confirmation.
- Given the user enters a name that already exists in the active raffle, when they submit, then a validation message is displayed (e.g., "That name is already entered") and the entry is rejected.
- Given the user submits an empty or whitespace-only name, when the form validates, then a validation message is displayed and the submission is blocked.
- Given no raffle is active, when the entry page loads, then a message is displayed (e.g., "No raffle is currently active") and no form is rendered.
- Given the active raffle has already started drawing, when the entry page loads, then a message is displayed (e.g., "Entries are closed") and no form is rendered.
- The name input shall be limited to 100 characters.
- The page shall be fully functional and usable on viewports as small as 320px wide.

---

### L2-040: Self-Registration Name Validation and Sanitization

**Traces to:** L1-012, L1-007

All participant names submitted through the self-registration page shall be validated and sanitized on both the client and server to prevent injection attacks and ensure data quality.

**Acceptance Criteria:**

- Given a name containing HTML tags or script content is submitted, when the server processes it, then the input is sanitized and stored as plain text only.
- Given a name exceeding 100 characters is submitted, when the server validates it, then the submission is rejected with an appropriate error message.
- Given a name containing only whitespace or empty string is submitted, when the server validates it, then the submission is rejected.
- The server shall trim leading and trailing whitespace from submitted names before duplicate checking and storage.
- The server shall validate all self-registration input independently of client-side validation to prevent bypass.

---

### L2-041: Self-Registration API Endpoint

**Traces to:** L1-012, L1-007, L1-010

The server shall expose a public API endpoint for participant self-registration that adds the submitted name to the active raffle's participant list.

**Acceptance Criteria:**

- Given a valid POST request with a name in the request body, when the active raffle is accepting entries, then the name is added to the raffle's participant list and a 201 response is returned.
- Given a POST request with a duplicate name, when the server processes it, then a 409 Conflict response is returned with a descriptive error message.
- Given a POST request when no raffle is active, then a 404 response is returned with a descriptive error message.
- Given a POST request when the active raffle has already started drawing, then a 403 response is returned with a descriptive error message indicating entries are closed.
- The endpoint shall not require authentication (it is public).
- The endpoint shall enforce rate limiting to prevent abuse (maximum 10 requests per IP per minute).
- The endpoint shall be protected against CSRF by requiring a JSON content type.

---

## Raffle Entry Lifecycle

### L2-042: Entry Window State Tracking

**Traces to:** L1-013, L1-004

The system shall track whether an active raffle is in the "accepting entries" state or the "drawing started" state. The transition from accepting entries to drawing started shall occur automatically when the first name is drawn and shall be irreversible (until the raffle is reset).

**Acceptance Criteria:**

- Given a raffle is activated and no names have been drawn, when the system checks the entry state, then the raffle is in the "accepting entries" state.
- Given a raffle is active and accepting entries, when the first draw occurs, then the raffle transitions to the "drawing started" state.
- Given a raffle is in the "drawing started" state, when checked again, then the raffle remains in the "drawing started" state regardless of further actions (until reset).
- Given an administrator resets a raffle's draws (per L2-009), when the reset completes, then the raffle returns to the "accepting entries" state.
- The entry state shall be derived from whether any draws have occurred (i.e., no separate manual toggle is required).

---

### L2-043: Block Entry After Drawing Starts

**Traces to:** L1-013, L1-010

Once the first name has been drawn from a raffle, the self-registration API shall reject all new entry attempts for that raffle. This enforcement shall occur at the server level regardless of client-side state.

**Acceptance Criteria:**

- Given a raffle has had at least one name drawn, when a self-registration POST request is received, then the server returns a 403 response with the message "Entries are closed."
- Given two concurrent requests — one to draw a name and one to self-register — when the draw completes first, then the self-registration request is rejected.
- The server shall use database-level checks (not in-memory state) to enforce this constraint, ensuring consistency across server restarts and multiple instances.

---

### L2-044: QR Code Page Responsiveness

**Traces to:** L1-012, L1-006

The QR code display page shall be fully responsive across all viewport sizes.

**Acceptance Criteria:**

- Given a viewport width < 576px (xs), when the QR code page loads, then the QR code is displayed at a minimum of 200×200 CSS pixels, centered, with instructional text stacked above or below.
- Given a viewport width ≥ 576px and < 768px (sm), when the QR code page loads, then the QR code is displayed at a minimum of 240×240 CSS pixels, centered.
- Given a viewport width ≥ 768px (md and above), when the QR code page loads, then the QR code is displayed at a minimum of 256×256 CSS pixels, centered, with instructional text adjacent.
- Given a viewport width ≥ 1200px (xl), when the QR code page loads, then the QR code scales up to a maximum of 400×400 CSS pixels for projection/large-screen readability.

---

### L2-045: Participant Entry Page Responsiveness

**Traces to:** L1-012, L1-006

The participant self-registration page shall be fully responsive and optimized for mobile-first usage.

**Acceptance Criteria:**

- Given a viewport width < 576px (xs), when the entry page loads, then the name input and submit button are displayed full-width with touch-friendly sizing (minimum 44×44 CSS pixels for the button).
- Given a viewport width ≥ 576px (sm), when the entry page loads, then the form is centered with a maximum width of 480px.
- Given a viewport width ≥ 768px (md and above), when the entry page loads, then the form is centered within a card-like container with comfortable spacing.
- The form shall remain usable and visually coherent on viewports as narrow as 320px.

---

### L2-046: Self-Registration Accessibility

**Traces to:** L1-012, L1-008

The QR code page and participant self-registration page shall conform to WCAG 2.2 Level AA accessibility standards.

**Acceptance Criteria:**

- The QR code image shall include a descriptive alt text (e.g., "QR code to enter the raffle").
- The name input field shall have a visible label associated via the `for`/`id` pattern.
- All form validation messages shall be announced to screen readers using an ARIA live region.
- The confirmation and error states shall be conveyed through text, not color alone.
- The submit button shall be keyboard-accessible and have a visible focus indicator.
- The QR code page shall provide a text link as an alternative to scanning (the direct URL of the entry page) for users who cannot scan QR codes.
